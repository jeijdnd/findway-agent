# 关键决策记录

> 记录项目中每个重要决策及其原因，方便回溯"为什么这么做"

## 决策时间线

### [2026-05-22] 决策：技术栈选择
- **背景**：项目需要选择前端框架、后端框架、桌面壳方案
- **选项**：
  - A) React + Vite + FastAPI（当前选择）
  - B) Vue + Webpack + Flask
  - C) Angular + Electron + Django
- **最终选择**：A) React + Vite + FastAPI
- **原因**：React生态成熟，Vite开发体验好，FastAPI性能高且适合API开发
- **影响**：整个项目的技术架构

### [2026-05-22] 决策：数据存储方案
- **背景**：项目需要存储配置、项目数据、聊天历史等
- **选项**：
  - A) JSON文件（当前选择）
  - B) SQLite数据库
  - C) PostgreSQL数据库
- **最终选择**：A) JSON文件
- **原因**：对初学者友好，可以直接用记事本打开查看和修改，无需数据库知识
- **影响**：数据持久化方式，所有API需要处理JSON文件读写

### [2026-05-22] 决策：AI对话实现方式
- **背景**：需要实现AI对话功能，驱动项目操作
- **选项**：
  - A) 关键词意图识别（V1当前实现）
  - B) LLM API调用（V2目标）
  - C) 本地模型（Ollama等）
- **最终选择**：A) 关键词意图识别（V1） → B) LLM API调用（V2）
- **原因**：V1先用关键词识别快速实现基本功能，V2升级为真正的LLM驱动
- **影响**：V1的AI对话功能有限，V2需要重构为LLM调用

### [2026-05-22] 决策：桌面应用打包方案
- **背景**：V2需要将Web应用包装为桌面应用
- **选项**：
  - A) Electron（当前选择）
  - B) Tauri
  - C) PyInstaller（仅Python）
  - D) 纯Web应用（不打包）
- **最终选择**：A) Electron
- **原因**：V1已有React前端，Electron零成本接入；Python FastAPI后端通过child_process.spawn拉起；用户使用Windows系统，Electron兼容性最佳；Electron生态成熟，有系统托盘、自动更新、原生通知支持
- **影响**：V2 T01任务的实现，需要创建Electron主进程和预加载脚本

### [2026-05-24] 决策：V2升级策略
- **背景**：V1功能已完成，需要规划V2升级路径
- **选项**：
  - A) 一次性重构所有功能
  - B) 渐进式升级，保持V1兼容
  - C) 先做桌面壳，再逐步升级其他功能
- **最终选择**：C) 先做桌面壳，再逐步升级其他功能
- **原因**：降低风险，用户能尽快体验到桌面应用；V1功能已稳定，可以作为V2的基础
- **影响**：V2 T01作为第一个V2任务，完成后用户即可使用桌面应用

## 已确认的技术决策

### WPS集成方案
- **推荐方案**：COM接口自动化（通过pywin32调用WPS COM对象）
- **降级方案**：文件直接读取（使用openpyxl读取.xlsx/.et文件）
- **前提**：用户已用WPS打开简易清单文件（COM接口）或保存文件后拖入（文件读取）

### 格原协同平台接入
- **推荐方案**：如果格原提供Web API或数据库直连，Agent通过HTTP请求拉取项目列表
- **降级方案**：模仿用户操作，通过网页抓取或读取格原本地缓存文件获取项目信息
- **触发方式**：定时轮询（可配置间隔）+ 手动刷新按钮

### API Key管理
- **当前阶段**：先用模拟回复，不接真实LLM
- **后续实现**：API Key放在环境变量（LLM_API_KEY），不写死在代码里
- **配置格式**：config.json中的llm配置节
