## 二、产品定义

### 2.1 V2 升级目标（3 个正交目标）

1. **桌面化与本地增强**：从浏览器页面升级为独立桌面窗口应用，自动扫描本地项目目录，实现"打开即用"的桌面体验。
2. **真 AI 驱动交互**：从关键词意图识别升级为 LLM 驱动的多轮对话，所有复杂操作通过自然语言触发，支持多模型配置与按任务切换。
3. **生态打通**：接入 WPS（用户主力办公软件）和格原协同平台（公司项目管理系统），让 Agent 融入现有工作流而非额外工具。

### 2.2 用户故事

| # | 用户故事 |
|---|---------|
| US-1 | 作为标识设计师，我希望双击桌面图标就能打开 Agent（像微信一样），而不是打开浏览器输入 localhost，这样我可以更快开始工作。 |
| US-2 | 作为标识设计师，我希望 Agent 自动发现我电脑上的项目文件夹，不用我一个一个手动创建项目卡片，因为我有几十个项目散落在不同目录。 |
| US-3 | 作为标识设计师，当公司协同平台分配新项目给我时，我希望 Agent 自动获取项目基本信息（名称、类型、楼栋），不用反复切换平台查看。 |
| US-4 | 作为标识设计师，我在 WPS 里写的简易清单（可能格式不规范），希望 Agent 能自动识别并映射到标准标识清单模板，不用我重新逐行录入。 |

## 三、技术规范

### 3.1 需求池（按优先级）

#### P0 — 必须交付（V2 核心）

| 编号 | 需求 | 说明 |
|------|------|------|
| **P0-1** | Electron 桌面壳 | 用 Electron 打包现有 React 前端，启动时自动拉起 Python FastAPI 后端，系统托盘驻留，窗口记忆位置/大小 |
| **P0-2** | LLM 对话引擎替换 | 将 V1 关键词意图识别替换为 OpenAI 兼容 API 调用，支持多轮上下文对话，实现流式输出（打字动画） |
| **P0-3** | 多 API 配置面板 | 设置页面支持添加/编辑/删除多个 LLM API（名称、Endpoint、API Key、模型名），对话中可下拉切换当前模型 |
| **P0-4** | 本地目录扫描与自动发现 | 配置监控目录 → 递归扫描 → 识别项目文件夹（根据特征：含清单 xlsx、DWG 文件等）→ 自动注册到项目列表 |

#### P1 — 应该交付

| 编号 | 需求 | 说明 |
|------|------|------|
| **P1-1** | WPS 清单识别与映射 | 通过 COM 接口读取 WPS 中已打开的简易清单 → AI 推断列映射关系 → 自动生成标准清单 → 用户确认后保存 |
| **P1-2** | 格原协同平台接入 | 通过 API 或网页抓取获取新项目列表 → 自动创建项目卡片 → 提醒用户 |
| **P1-3** | 按任务自动切换模型 | 根据用户意图自动选择模型（创意→模型A，规范→模型B），用户可在设置中配置映射规则 |
| **P1-4** | 对话驱动全功能 | 确保所有 V1 功能（项目 CRUD、清单对比、旧项目匹配、规范查询）均可通过对话触发 |

#### P2 — 锦上添花

| 编号 | 需求 | 说明 |
|------|------|------|
| **P2-1** | WPS 清单同步回写 | 在 Agent 中修改清单后，自动同步更新回 WPS 中打开的文件 |
| **P2-2** | 系统通知集成 | 新项目分配、清单对比完成等事件通过 Windows 原生通知提醒 |
| **P2-3** | 离线降级对话 | 无网络时自动切换到本地规则引擎，保证基础功能可用 |

### 3.3 技术架构决策

#### 3.3.1 Electron 打包方案

```
┌──────────────────────────────────┐
│         Electron Shell            │
│  ┌────────────┐ ┌─────────────┐  │
│  │  BrowserWindow │  Main Process │  │
│  │  (React 前端)  │  (Node.js)    │  │
│  │  localhost:5173 │              │  │
│  └────────────┘ └──────┬──────┘  │
│                        │          │
│            spawn Python 子进程     │
│                        │          │
│            ┌───────────▼────────┐ │
│            │  Python FastAPI     │ │
│            │  localhost:8765     │ │
│            └────────────────────┘ │
└──────────────────────────────────┘
```

选择 Electron 而非 Tauri 的理由：
- V1 已有 React 前端，Electron 零成本接入
- Python FastAPI 后端通过 `child_process.spawn` 拉起，Tauri 同样需要额外进程管理
- 用户使用 Windows 系统，Electron 在 Windows 上兼容性最佳
- Electron 生态成熟的系统托盘、自动更新、原生通知支持

#### 3.3.2 WPS 集成：双路径方案

**路径 A（推荐，P1）：COM 接口自动化**
- 通过 `pywin32` (`win32com.client`) 调用 WPS COM 对象
- 连接已打开的 WPS 表格实例：`win32com.client.GetObject(Class='Et.Application')`
- 读取活动工作表的全部数据，获取行列范围
- 将原始数据交给 LLM 推断列映射关系
- **前提**：用户已用 WPS 打开简易清单文件

**路径 B（降级，P1 备选）：文件直接读取**
- 使用 `openpyxl` 直接读取 `.xlsx/.et` 文件
- 用户将 WPS 文件保存后，拖入或选择文件给 Agent
- 优点：不依赖 WPS 运行状态，更稳定
- 缺点：多一步"保存"操作

**映射流程（两种路径共用）**：
```
简易清单（非标准列名）          标准标识清单模板
┌──────────────┐            ┌──────────────────┐
│ 牌子名称      │  ──AI推断──→ │ 名称              │
│ 大小          │  ──AI推断──→ │ 参考尺寸           │
│ 用什么材料     │  ──AI推断──→ │ 参考材质           │
│ 要多少        │  ──AI推断──→ │ 数量              │
│ 备注/说明     │  ──AI推断──→ │ 备注              │
│ (缺失)       │  ──补空──→   │ 编号 / 单位 / 是否带电 │
└──────────────┘            └──────────────────┘
```

> **可行性结论**：WPS COM 接口在 Windows 上完全可行，`pywin32` + `win32com.client` 可以读写 WPS 表格。主要风险是 WPS 版本差异（个人版/专业版 COM 注册不同）和非标准清单格式的识别准确率。建议先实现路径 B 保证基本可用，路径 A 作为增强。

### 3.4 格原协同平台接入方案

用户提到"格原协同平台"，这是 AutoCAD 插件生态中的协同管理系统。接入策略：

- **方案 A**（推荐）：如果格原提供 Web API 或数据库直连，Agent 通过 HTTP 请求拉取项目列表
- **方案 B**（降级）：模仿用户操作，通过网页抓取或读取格原本地缓存文件获取项目信息
- **触发方式**：定时轮询（可配置间隔）+ 手动刷新按钮

> **注意**：此项需用户确认格原平台的接口能力，属于外部依赖。

## 四、待确认问题（Open Questions）

| # | 问题 | 重要性 | 建议决策 |
|---|------|--------|---------|
| Q1 | **Electron vs Tauri 最终确认？** 当前建议 Electron（与 V1 React 前端无缝衔接），但 Tauri 体积更小。 | 高 | 建议 Electron，后续可迁移 |
| Q2 | **WPS 版本？** 用户用 WPS 个人版还是专业版？COM 接口注册名可能不同（`Et.Application` vs `Ket.Application` vs `WPS.Application`）。 | 高 | 需用户确认版本后实测 |
| Q3 | **格原协同平台是否有 API？** 如果没有 API，需要降级到网页抓取或本地文件读取，可靠性和时效性会下降。 | 高 | 需用户提供格原平台技术对接人 |
| Q4 | **LLM API 费用预算？** 多模型意味着多路 API 调用成本。DeepSeek 当前最便宜，通义千问有免费额度。 | 中 | 建议首批支持 DeepSeek + 通义千问 |
| Q5 | **简易清单格式多样性？** 用户的"简易清单"到底有多不标准？列名、行数、合并单元格情况？ | 中 | 需用户提供 3-5 个真实简易清单样本 |
| Q6 | **本地项目目录结构？** 用户的旧项目文件夹有统一的命名和结构规范吗？还是没有规律？ | 中 | 需用户提供 2-3 个典型项目目录截图 |
| Q7 | **是否需要自动更新机制？** 桌面应用是否需要自动检测版本更新并提示升级？ | 低 | 建议 V2.1 再做 |
| Q8 | **多窗口支持？** 是否需要像微信一样支持多个独立窗口（如一个看对话一个看清单）？ | 低 | 建议 V2 不做，保持单窗口 |
